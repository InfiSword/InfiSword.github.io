Architecture Redesign

1. UI(RectTransform)에서 GameObject(Transform) 기반으로의 전환

초기 단계에서는 윈도우 바탕화면의 느낌을 구현하고자 파일 유닛을 UI 시스템으로 구축했습니다. 하지만 프로젝트 규모가 커짐에 따라 다음과 같은 결정적인 설계적 한계를 마주했습니다.

  • 좌표계 종속성: RectTransform은 캔버스의 앵커(Anchor)와 피벗(Pivot) 설정에 따라 월드 좌표값이 상대적으로 변합니다. 이로 인해 유닛의 절대적인 위치를 파악하는 로직이 매우 복잡해졌습니다.
  • 연산 비용의 증가: 월드 좌표 기반의 바이러스(적)가 UI 기반의 파일 유닛 간의 거리 계산 및 물리적 충돌 판정 로직을 짜고자 할 때, 파일을 타겟팅하거나 파티클 이펙트를 출력할 때마다 Camera.WorldToScreenPoint, Camera.ScreenToWorldPoint 등의 변환 함수를 매 프레임 호출해야 했습니다.
  • 계층 구조의 복잡성: 서로 다른 부모 객체(UI Hierarchy) 간의 위치 변환 로직은 코드의 가독성을 해치고 복잡한 로직의 구조가 나타났습니다.

이를 해결하기 위해 모든 유닛을 GameObject(Transform) 기반으로 전면 교체하여 월드 좌표계로 통일하여, 성능과 확장성, 로직을 간단하게 짜기 위한 이점들을 동시에 확보했습니다.

2. 중앙 집중식 입력 시스템 (InputManager)

수많은 유닛이 개별적으로 이벤트를 수신하지 않고, InputManager가 레이캐스트를 통해 입력을 통제하도록 설계했습니다.

// InputManager.cs: Raycast를 통한 중앙 집중식 상호작용 객체 탐지 private IInteractable GetInteractableUnderMouse() { // 레이캐스트를 사용하여 월드 좌표상의 유닛 검출 RaycastHit2D hit = Physics2D.Raycast(m_mouseWorldPosition, Vector2.zero, 0f, unitLayerMask); if (hit.collider != null) { return hit.collider.GetComponent<IInteractable>(); } return null; }
